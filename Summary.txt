Concrete Implementation Plan For Recommendation Review Freshness And Docs-Grounding

Goal

- Close the executable review-freshness gap without weakening apply-time safety.
- Keep `pattern` browse-only.
- Land the smallest implementation that makes server-only prompt-shaper drift visible on the review surfaces that actually need it.

Recommended Scope Decision

- Implement hybrid client plus server review freshness for:
  - `template`
  - `template-part`
  - `global-styles`
  - `style-book`
- Keep `block` client-review-only in this pass.
- Keep apply-time `resolvedContextSignature` plus `resolveSignatureOnly` unchanged as the mutation safety contract for every executable surface, including `block`.

Why This Scope

- The medium-severity gap is on the review-before-apply surfaces, so that should be the first cut.
- `block` already has direct-apply client freshness plus apply-time server revalidation; its remaining issue is docs-guidance drift only.
- Deferring block review freshness avoids adding a second async freshness loop to the inspector while the shared server-review path is still being introduced.

Recommended Contract Decision

- Add a new server-issued `reviewContextSignature`.
- Do not redefine `resolvedContextSignature`.
- Build `reviewContextSignature` from a normalized prompt-shaper digest, not from the entire raw resolved context.
- Return `reviewContextSignature` on:
  - full recommendation responses for `template`, `template-part`, `global-styles`, and `style-book`
  - signature-only revalidation responses on those same routes

Why Prompt-Shaper Digest Instead Of Raw Context

- The prompt builders already cap and normalize the inputs that the model can actually see.
- Hashing prompt-visible server-only inputs avoids false drift from backend fields that never reach the prompt.
- The digest can stay deterministic by reusing the same associative-array normalization rules already used by `RecommendationResolvedSignature`.

Implementation Shape

1. Add a dedicated review-signature helper.

- Add `inc/Support/RecommendationReviewSignature.php`.
- Reuse the same normalization semantics as `inc/Support/RecommendationResolvedSignature.php`:
  - recursively normalize objects to arrays
  - `ksort()` associative arrays
  - preserve list order
  - hash a JSON payload with `sha256`
- Keep the helper name explicit so the codebase does not blur review freshness and apply freshness.

2. Add per-surface prompt-shaper digest builders on the server.

- In `inc/Abilities/TemplateAbilities.php` add private helpers for:
  - building the template review-signature payload
  - building the template-part review-signature payload
- In `inc/Abilities/StyleAbilities.php` add private helpers for:
  - building the global-styles review-signature payload
  - building the style-book review-signature payload
- Do not add a block review-signature payload in this pass.

The review-signature payloads should include only server-only prompt shapers that are not already covered by the client signature.

- `template`
  - prompt-visible candidate `patterns`
  - prompt-visible `themeTokens`
  - docs-guidance digest
- `template-part`
  - prompt-visible candidate `patterns`
  - prompt-visible `themeTokens`
  - docs-guidance digest
- `global-styles`
  - prompt-visible `themeTokens`
  - docs-guidance digest
- `style-book`
  - prompt-visible `themeTokens`
  - prompt-visible `blockManifest` fields actually emitted by `StylePrompt::build_user()`
    - `supports`
    - `inspectorPanels`
  - docs-guidance digest

Docs-guidance digest rules

- Limit to the same chunk count the prompt uses today.
- Normalize to a bounded shape such as:
  - `id`
  - `title`
  - `url`
  - `excerpt`
- Do not hash raw transport metadata or cache keys.

3. Thread `reviewContextSignature` through the ability responses.

- Update `inc/Abilities/TemplateAbilities.php`
  - `recommend_template()`
  - `recommend_template_part()`
- Update `inc/Abilities/StyleAbilities.php`
  - `recommend_style()`
- Compute `reviewContextSignature` after docs guidance is collected, but before model execution.
- Full responses should include both:
  - `reviewContextSignature`
  - `resolvedContextSignature`
- Signature-only responses should also include both fields for the four review surfaces.
- Leave `inc/Abilities/BlockAbilities.php` alone for now.

4. Update REST handlers and ability schemas.

- Update `inc/REST/Agent_Controller.php`
  - `handle_recommend_template()`
  - `handle_recommend_template_part()`
  - `handle_recommend_style()`
- No route-shape change is needed beyond passing through the new field.
- Update `inc/Abilities/Registration.php` output schemas so the affected abilities expose `reviewContextSignature`.
- Keep the current block envelope and block schema untouched in this pass.

5. Add a shared client-side server-review revalidation path.

- Extend `src/store/executable-surface-runtime.js` with a new shared helper for review freshness revalidation.
- Reuse the existing lightweight server POST path with `resolveSignatureOnly: true`.
- Compare:
  - stored `reviewContextSignature`
  - freshly resolved `reviewContextSignature`
- Only run this review revalidation when:
  - the stored result still matches the current client request signature
  - the surface has a stored server review signature
  - the surface is one of the four hybrid-review surfaces

Suggested helper shape

- `createExecutableSurfaceReviewFreshnessAction(...)`
- one async thunk per executable review surface
- request tokening or result-token gating so stale async completions are ignored

6. Add store state for review freshness without overloading apply freshness.

- Update `src/store/index.js` state, actions, reducer, and selectors for the four review surfaces.
- Add new state fields per surface:
  - `reviewContextSignature`
  - `reviewFreshnessStatus` or `reviewFreshnessCheckedAt`
  - `reviewStaleReason`
- Keep the existing apply-state fields as-is:
  - `*ApplyStatus`
  - `*ApplyError`
  - `*StaleReason` if it stays apply-specific

Recommended reason model

- `client`
- `server-review`
- `server-apply`

If the existing `*StaleReason` fields remain tied to apply state, then:

- compute client review staleness locally
- store server review staleness in the new review freshness fields
- map both into one `effectiveStaleReason` in the components

7. Thread the new field through fetch success paths.

- Update `src/store/index.js` dispatch helpers:
  - `dispatchTemplateRecommendations()`
  - `dispatchTemplatePartRecommendations()`
  - `dispatchGlobalStylesRecommendations()`
  - `dispatchStyleBookRecommendations()`
- Update the matching action creators:
  - `setTemplateRecommendations()`
  - `setTemplatePartRecommendations()`
  - `setGlobalStylesRecommendations()`
  - `setStyleBookRecommendations()`
- Persist `reviewContextSignature` beside the existing `contextSignature` and `resolvedContextSignature`.
- Clear server-review stale state on:
  - new loading state
  - fresh fetch success
  - explicit clear actions

8. Hook review freshness revalidation into the review surfaces.

- Update:
  - `src/templates/TemplateRecommender.js`
  - `src/template-parts/TemplatePartRecommender.js`
  - `src/global-styles/GlobalStylesRecommender.js`
  - `src/style-book/StyleBookRecommender.js`
- Keep the current local client-signature comparison exactly as the first stale signal.
- After the client signature still matches, trigger the async server-review revalidation.
- Compute `effectiveStaleReason` in this order:
  - client request-signature drift
  - server review-signature drift
  - server apply drift from apply-state

UI copy changes

- Keep the current client-drift messaging.
- Add distinct review-copy for server review drift:
  - "This result no longer matches the current server review context..."
- Keep separate apply-copy for server apply drift:
  - "This result no longer matches the current server-resolved apply context..."

9. Leave apply-time safety intact.

- Do not change `guardSurfaceApplyResolvedFreshness()` except to tolerate extra fields in the signature-only response.
- Keep the existing order:
  - client request-signature guard first
  - `resolveSignatureOnly` apply revalidation second
  - mutation only after both checks pass

Proposed Rollout Order

1. Add PHP helper and server payload builders.
2. Return `reviewContextSignature` from template/template-part/style abilities and REST/schema layers.
3. Add store state plus shared review revalidation helper.
4. Wire the four review surfaces to async server-review revalidation.
5. Land PHPUnit coverage for docs grounding and signature drift.
6. Land JS store and component coverage.
7. Update docs last, once field names and response shapes are stable.

Out Of Scope For This Pass

- Adding review freshness revalidation to `block`
- Changing `pattern`
- Collapsing review freshness and apply freshness into one field
- Widening family cache keys with raw pattern-name or override-name lists
- Making docs guidance part of the mutation validator or executor

Definition Of Done

- `template`, `template-part`, `global-styles`, and `style-book` expose and persist `reviewContextSignature`.
- Those four surfaces can become stale from server-only prompt-shaper drift even when the client signature still matches.
- Apply-time mutation safety still relies on `resolvedContextSignature` and `resolveSignatureOnly`.
- `block` is explicitly documented and tested as client-review-only plus server-apply-safe.
- Docs-grounding coverage is extended for template and template-part bounded-scope behavior.
- Historical docs no longer contradict the implemented freshness contract.
