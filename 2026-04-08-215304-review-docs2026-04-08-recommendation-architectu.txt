  A few concerns worth challenging:

  Finding 1 risks being self-contradictory. It says the audit "overstates a
  freshness gap" that was already closed, but the audit is supposed to describe
  the shipped system. If the shipped system already has resolvedContextSignature
   + resolveSignatureOnly, the Finding should cite specific lines in the audit
  doc that misstate it, not just note the gap was closed. Otherwise it reads as
  "the audit was wrong" rather than "the audit needs a correction."

  Finding 3 (live-structure overlays) and Finding 4 (Global Styles / Style Book 
  shell) are reversed in priority. The style surfaces are the lowest-risk
  extraction ‚ÄĒ they already share the same operation core, the same signature
  builder, and the same REST endpoint. Extracting a StyleSurfacePanel wrapper is
   a pure JS UI refactor with no backend touch. The template live-structure work
   involves PHP overlay logic in TemplateAbilities and parser validation in
  TemplatePrompt::parse_response() ‚ÄĒ that's higher risk and belongs later. The
  phased plan puts style first (Phase 3) after the store runtime (Phase 2),
  which is reasonable, but Finding 4 should acknowledge this is the easiest win.

  The "Simplify Now" list conflates documentation changes with code changes.
  Finding 1 is almost entirely a docs/naming fix ‚ÄĒ making the two-step freshness
   contract explicit in how the architecture is described. That doesn't need a
  phased plan at all; it just needs someone to rename guardSurfaceApplyFreshness
   vs guardSurfaceApplyResolvedFreshness in the write-up and update the
  validation checklist. The phased plan correctly isolates this as "normalize
  and document" in Phase 1, but the "Simplify Now" framing suggests these are
  all code-level changes when some are just description-level corrections.

  The Validation Checklist is doing double duty ‚ÄĒ it's both regression tests for
   the phased refactors and a retrospective accuracy check against the audit's
  own claims. Those should probably be separate: one list verifying the phased
  extraction work, one list verifying the audit's factuality against the current
   codebase. Running the "verify block keeps { payload, clientId } envelope"
  check against the actual Agent_ControllerTest would immediately confirm
  whether Finding 1's "already shipped" claim holds.

  One thing the doc doesn't address: the cost of the extraction phases vs. the 
  benefit. Five phases is a meaningful investment. The audit correctly
  identifies duplication, but doesn't quantify how much maintenance burden the
  duplication actually creates today. If the answer is "it's annoying but hasn't
   caused bugs," that's a different priority than "the duplication has caused N
  regressions and blocks feature work." The phased plan reads as if the
  refactors are obviously worth doing, but the reader has to take that on faith.

  Verdict: Direction is sound. The main risks are (1) treating Finding 1 as a
  code refactor when it's mostly a documentation correction, (2) not separating
  "verify the audit is accurate" from "execute the refactor plan," and (3)
  understating that the style shell (Phase 3) is the easiest extraction and
  arguably the best first candidate for someone to prototype and validate the
  whole approach.
