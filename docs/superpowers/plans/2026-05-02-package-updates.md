# Package Updates Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Follow the execution protocol in [../reference/agentic-plan-implementation-guide.md](../reference/agentic-plan-implementation-guide.md). Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring `package.json` and `package-lock.json` to current versions in four bisectable commits, deferring `composer.json` PHPCS bump (upstream-blocked) and capturing baselines so any regression can be attributed to one stage.

**Architecture:** Linear sequence on `master`: baseline (no commit) → 11 named in-range bumps → Playwright pinned bump → six coupled WordPress 7.0-aligned majors → `@wordpress/scripts` toolchain → final full-verify with E2E. Per-package `npm install <pkg>@<exact-version>` everywhere; never `npm update`. Per-stage capture of build SHA256 + lockfile size under `output/verify/` so transitive churn surfaces as a numeric stop signal.

**Tech Stack:** Node 24 / npm 11 (`.nvmrc`), `@wordpress/scripts` 30 (build), Jest (unit), Playwright (E2E), `node scripts/verify.js` (aggregated harness writing `output/verify/summary.json`).

**Spec:** `docs/superpowers/specs/2026-05-02-package-updates-design.md`

---

## File Map

| File | Action | Responsibility |
|---|---|---|
| `package.json` | Modify | New version pins for 11 in-range packages (Stage 1), `@playwright/test` exact pin (Stage 2), six WP majors (Stage 3), `@wordpress/scripts` (Stage 4). |
| `package-lock.json` | Modify | Regenerated by each `npm install` invocation. Diff size watched per stage. |
| `output/verify/baseline-fast.json` | Create (Stage 0) | Fast-verify summary at HEAD before any change. |
| `output/verify/baseline-full.json` | Create (Stage 0) | Full-verify (with E2E) summary at HEAD. The bar the final verify must match-or-beat. |
| `output/verify/baseline-build.sha256` | Create (Stage 0) | SHA256 of `build/*.{js,asset.php,css}` at HEAD. |
| `output/verify/baseline-lockfile.txt` | Create (Stage 0) | `wc -l package-lock.json` + package count. |
| `output/verify/stage-<N>-build.sha256` | Create per stage | Post-stage build manifest for `diff` against previous. |
| `output/verify/stage-<N>-lockfile.txt` | Create per stage | Post-stage lockfile size + package count. |
| `src/**/*.js` | Modify only if forced | Stage 3 may force adjustments to Inspector / admin / context modules if WP majors removed APIs the plugin uses. Adjustments live in the same commit as the bump. |
| `composer.json`, `composer.lock` | UNCHANGED | PHPCS 3 → 4 deferred (WPCS upstream block). |

`output/verify/` is gitignored — baseline + per-stage artifacts persist on disk for the run but never get committed.

---

## Conventions used in every stage

- **Always** invoke from repo root `/home/dev/flavor-agent`.
- **Never** run `npm update` (per spec "Mechanism choice"); use `npm install <pkg>@<exact-version>` only.
- **Never** use `--legacy-peer-deps` or `--force` on `npm install`. A resolution failure is a stop signal, not a flag-tweak situation.
- **Never** use `git commit --no-verify`, `git commit --amend` on a previously committed stage, or `git push --force`.
- **Never** bundle multiple stages in one commit.
- After every `npm install`, run `npm ci` once as a sanity check that manifest and lockfile agree (it should succeed without modifying anything). If it modifies anything or fails: stop.
- Build manifest capture command (used in Tasks 0, 1, 2, 3, 4): `(cd build && find . -type f \( -name '*.js' -o -name '*.asset.php' -o -name '*.css' \) -print0 | xargs -0 sha256sum) | sort`
- Lockfile manifest capture command: `wc -l package-lock.json` and `node -e "const lf = require('./package-lock.json'); console.log('packages:', Object.keys(lf.packages).length)"`

## Reverting

If a single stage commit needs to be undone after the fact (e.g. discovered regression days later, isolated to one stage by `git bisect`):

```bash
git revert <commit-sha>
```

Each stage commit is self-contained (`package.json` + `package-lock.json` + any forced source edits) so a single `git revert` cleanly undoes one stage without touching others. Run `npm ci` after the revert to bring `node_modules/` back into sync with the reverted lockfile.

If multiple stages need reverting and the order matters, revert in reverse order (Stage 4 first, then 3, etc.) so peer-dep constraints don't conflict.
