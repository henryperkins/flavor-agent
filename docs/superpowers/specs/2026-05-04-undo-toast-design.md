# Undo Toast

## Background

The Flavor Agent design system bundle (May 2026) introduces a new component — the **Undo Toast** — to bridge the gap between silent auto-applied AI changes and the persistent `Settings > AI Activity` audit surface. Today, applies on the Inspector block surface and Global Styles surface complete without any post-apply confirmation; users must navigate to the activity log to discover what changed and to undo it.

The design system already provides:

- A complete `.fa-toast` CSS spec in the bundle's `components.css` (icon, message, detail, action, close, progress; success/warning/error/compact variants; dark background; teal countdown bar; rectilinear 2px corners).
- Anticipated reduced-motion handling in `src/tokens.css:142` that already targets `.flavor-agent-toast-region *`.

The token layer, brand colors, and motion tokens needed by the toast are already in `src/tokens.css`. No new tokens are required.

**Token + class namespace.** The design bundle uses `--fa-*` tokens and `.fa-*` class names (`--fa-color-ink`, `.fa-toast`). The codebase uses `--flavor-agent-*` and `.flavor-agent-*` (`--flavor-agent-color-ink`, `.flavor-agent-toast`). The implementation uses the **codebase namespace exclusively** — every CSS class is `.flavor-agent-toast*` and every token reference is `--flavor-agent-*`. The bundle's `--fa-*` names appear only in this spec when quoting source.

## Goals

1. Surface every executable AI apply with a non-disruptive toast that exposes the existing activity-log Undo affordance at the moment of change.
2. Match the design bundle's `.fa-toast` spec under the codebase's `flavor-agent-` namespace.
3. Reuse the canonical activity-log undo path so the toast and the `AIActivitySection` row Undo never diverge.

## Non-goals

- New REST routes.
- Changes to `Activity\Repository`, `Activity\Permissions`, or `Activity\Serializer`.
- Changes to `AIActivitySection` or its undo behavior.
- Toasts on Pattern recommendations (browse-only, no apply path) or Navigation (advisory-only).
- Persistent dismissal preferences. Reduced-motion is the only behavioral switch.

## Scope

### Surfaces that fire toasts

All executable apply paths:

| Surface | Apply action (verified name) | Toast title | Toast detail (mono, illustrative) |
|---|---|---|---|
| Block (attr-only) | `applySuggestion` (`src/store/index.js:1388`) | "Block updated" | `core/heading · attr=fontWeight` |
| Block (structural) | `applyBlockStructuralSuggestion` (`src/store/index.js:1596`) | "Block updated" | `core/group · insert pattern` |
| Template | `applyTemplateSuggestion` (generated from `EXECUTABLE_SURFACE_DEFS` baseName `Template`) | "Template applied" | `page-template-1 · 6 ops` |
| Template part | `applyTemplatePartSuggestion` (baseName `TemplatePart`) | "Template part applied" | `header · 3 ops` |
| Global Styles | `applyGlobalStylesSuggestion` (baseName `GlobalStyles`) | "Global styles updated" | `typography.fontSize · −0.04em` |
| Style Book | `applyStyleBookSuggestion` (baseName `StyleBook`) | "Style Book updated" | `core/button · primary` |

The four executor surfaces (`Template`, `TemplatePart`, `GlobalStyles`, `StyleBook`) share `buildExecutableSurfaceApplyThunk` in `src/store/executable-surface-runtime.js`; their apply actions are generated by `EXECUTABLE_SURFACE_DEFS` per-surface `methodNames.applySuggestion = "apply{baseName}Suggestion"` (`src/store/executable-surfaces.js:37`).

Pattern recommendations are browse-only and remain silent. Navigation is advisory and remains silent.

### Toast contents

```
[icon] [strong title] [mono detail]   [Undo button] [× close]
[───── teal/warning/error countdown bar ─────]
```

- **Icon:** success / warning / error variant per the design bundle.
- **Title:** sentence-case, ≤4 words, imperative-resultative ("Block updated").
- **Detail:** mono, ≤40 chars, technical anchor (block name, template slug + op count, style path + new value). Truncates with ellipsis if overflowed.
- **Undo:** button uses a disabled visual state with `title="Undo unavailable for this change"` when the apply succeeded but no activity entry id was returned. It remains focusable and in tab order with `aria-disabled="true"` and `tabIndex={0}` so the `Undo → Close` tab order is preserved; click/keyboard handlers must no-op when disabled.
- **Close:** button, always available.

### Behavior

- **Variant trigger matrix:**
  - `success` — every successful apply (the only enqueue source from apply paths).
  - `error` — undo failure only. The success toast transitions in place to the error variant and updates its content (see Failure of Undo below).
  - `warning` — reserved; no current trigger source. Apply failures stay inline in the recommendation panel (already surfaced there via `getBlockApplyError()` and equivalents); duplicating that as a toast would be noise.
- **Auto-dismiss:** 6s for `success`; 8s for `error`. Pause-on-hover **and** pause-on-focus-within pause both the countdown timer (`setTimeout`) and the countdown bar (CSS animation `play-state: paused`); resume on mouse-leave + focus-out. Both interactions matter: a keyboard-only user must be able to reach the Undo button via Tab without the timer firing first.
- **Keyboard reach (iframe surfaces).** The toast region is portaled to `document.body`, outside any Site Editor / Style Book iframe. Tab order from iframe content does not naturally cross into host-body overlays. To keep Undo reachable for keyboard-only users on iframe surfaces, the implementation registers a global `mod+shift+z` shortcut (via `@wordpress/keyboard-shortcuts`) that focuses the Undo button of the newest visible toast. `mod+z` itself stays bound to browser/native undo (we cannot intercept that consistently across iframes). The shortcut is no-op when the queue is empty. The shortcut is registered once, alongside the `<ToastRegion />` mount.
- **Stack:** max 3 visible. Queue is ordered oldest → newest; renders newest at the bottom (normal `column` flow, append-at-end). When a 4th arrives:
  - Evict the oldest **non-interacted** toast (not focused, not hovered).
  - If all 3 visible toasts are interacted, drop the incoming toast (do not backlog — the canonical record is in `Settings > AI Activity`).
- **A11y:** `role="status"`, `aria-live="polite"` (non-interrupting). Tab order: Undo → Close. Escape dismisses the focused toast.
- **Reduced motion:** the existing `prefers-reduced-motion` rule in `src/tokens.css:142` zeroes animation/transition durations on `.flavor-agent-toast-region *`. Because that rule would otherwise collapse the 6s progress animation to an instant jump-to-empty, the new toast CSS also adds `.flavor-agent-toast__progress { display: none; }` inside the same `prefers-reduced-motion` block. **Auto-dismiss is also disabled entirely under `prefers-reduced-motion`** (WCAG 2.2.1 — Timing Adjustable): without the visual countdown, a time-based dismissal a user can't perceive is hostile. The toast persists until the user clicks Undo or Close. The component reads `window.matchMedia('(prefers-reduced-motion: reduce)').matches` at mount and skips scheduling `setTimeout` when true; it also subscribes to the media-query change event so a mid-session preference change takes effect on the next enqueue. Hover/focus pause is irrelevant in this mode (no timer to pause).
- **Failure of Undo:** the toast stays visible while Undo runs. On success, the toast self-dismisses. On failure, the toast transitions in place to the `error` variant and updates its content to include the action context (`title`, `detail`, `entry id`) plus a short error hint. The 8s error timer starts at the moment of transition.

## Architecture

### New files

- `src/components/UndoToast.js` — **pure presentational**. Props: `{ id, variant, title, detail, undoLabel, onUndo, onDismiss, undoDisabled, onInteractionChange, autoDismissMs }`. Renders the icon, message, action, close, and progress bar. Hover **and** focus-within toggle the `is-paused` class on the progress bar and call `onInteractionChange(true|false)` so the region can flag the queue entry as interacted (eviction protection). The component owns *only* the `setTimeout` for auto-dismiss (a side-effect of its lifecycle, not durable state) and reads `prefers-reduced-motion` to decide whether to schedule the timer at all. **Variant transitions are not internal** — they arrive via props from `updateToast(id, patch)` in the slice. This keeps the component stateless w.r.t. the toast model.
- `src/components/ToastRegion.js` — `createPortal(document.body)` mount. Subscribes to the store's toast queue; renders up to 3 `<UndoToast>` instances in oldest→newest order; wires `onDismiss`, `onUndo`, and `onInteractionChange` to dispatch. Registers the global `mod+shift+z` shortcut on mount via `@wordpress/keyboard-shortcuts` (focus newest visible toast's Undo button; no-op if queue empty); unregisters on unmount.
- `src/store/toasts.js` — slice with:
  - actions: `enqueueToast(toast)`, `updateToast(id, patch)`, `dismissToast(id)`, `dismissAllToasts()`, `markToastInteracted(id, interacted)`
  - selector: `getToasts()`
  - helper: `buildToastForActivity({ surface, persistedEntry, suggestion, extras })` (single source of truth for surface→title and persisted-entry→activity-id mapping; see "Helper location" above)
  - reducer caps the queue at 3 with **non-interacted FIFO eviction**: enqueueing a 4th evicts the oldest entry whose `interacted === false`; if every visible entry is interacted, the new entry is dropped.

### CSS additions to `src/editor.css`

A new "Undo Toast" section at the end of the file. Mirrors the design's `.fa-toast` spec under the `flavor-agent-toast` namespace:

- `.flavor-agent-toast-region` — fixed-position bottom-center container, `pointer-events: none`, `flex-direction: column`.
- `.flavor-agent-toast` — dark background (`--flavor-agent-color-ink`), white text, 2px corners, 44px min-height, drop shadow, fade-in animation.
- `.flavor-agent-toast__icon` (+ `--success`, `--warning`, `--error` color variants).
- `.flavor-agent-toast__msg` containing `<strong>` title and `.flavor-agent-toast__detail` (mono, 60% white, ellipsized).
- `.flavor-agent-toast__action` — ghost-bordered button; sentence-case "Undo". Re-enable hit testing with `pointer-events: auto` and keyboard focus styles.
- `.flavor-agent-toast__close` — icon-only ghost button with `pointer-events: auto`.
- `.flavor-agent-toast__progress` — 2px hairline at the bottom; teal default; warning/error variants change color; `.is-paused` halts the animation.

### Wiring — in-action enqueue (no return-contract changes)

Each apply action already calls `await logStoreActivityEntry(...)` (or `await recordActivityEntry(...)` for executor surfaces), and `recordActivityEntry` in `src/store/activity-session.js:518` already returns the persisted entry (with its server `.id`). Today the return value is discarded; the apply action then dispatches `set*ApplyState('success', ...)` and `return true`.

The implementation **captures that return value** and dispatches `enqueueToast` from inside the same action, before returning. All call sites build the payload through one shared helper (see "Helper location" below):

```
const persisted = await logStoreActivityEntry(localDispatch, select, entry);
localDispatch(actions.setBlockApplyState(clientId, 'success', ...));
localDispatch(actions.enqueueToast(buildToastForActivity({
    surface: 'block',
    persistedEntry: persisted,
    suggestion,
    extras: { blockContext },           // surface-specific render hint
})));
return true;
```

`buildToastForActivity` always returns `variant: 'success'` from this site (the only enqueue source from apply paths). The `error` variant is produced inside the toast component when undo fails (see Failure of Undo). The helper handles `persisted == null` by returning a payload with `activityId: null`, which renders the toast with a disabled Undo button per the Toast contents section.

**This deliberately does not change the apply action's return contract.** Apply actions still return `true`/`false` per existing convention. `useSuggestionApplyFeedback` (`src/inspector/use-suggestion-apply-feedback.js:67`) keeps its `didApply = await applySuggestion(...)` boolean check unchanged. Existing tests asserting `expect(result).toBe(true)` (`src/store/__tests__/store-actions.test.js:2657`) keep passing without modification. The toast is a **side effect** of the apply action, not part of its contract.

**Touch-points (verified action names):**

- `applySuggestion` in `src/store/index.js:1388` — the block attr-only apply.
- `applyBlockStructuralSuggestion` in `src/store/index.js:1596` — block structural apply.
- `applyTemplateSuggestion`, `applyTemplatePartSuggestion`, `applyGlobalStylesSuggestion`, `applyStyleBookSuggestion` — generated by `createExecutableSurfaceRuntimeActionCreators` in `src/store/executable-surfaces.js:617`. The shared apply pipeline is in `src/store/executable-surface-runtime.js:425` (`await recordActivityEntry(...)`); a single insertion at that line makes all four surfaces enqueue without per-surface code.

### Helper location (avoid drift)

The single source of truth for toast payload construction lives in **`src/store/toasts.js`** (the new slice file). It exports `buildToastForActivity({ surface, persistedEntry, suggestion, extras })` alongside the actions. Both edit sites — the two block actions in `src/store/index.js` and the executor pipeline in `src/store/executable-surface-runtime.js:425` — import this helper. The per-surface title and the detail-formatter dispatch (e.g. `surface: 'template'` → "Template applied" + `${slug} · ${ops.length} ops`) live inside the helper, not at the call sites. This is the explicit anti-drift answer to the reviewer's question: there is exactly one file that knows how to translate an activity entry + suggestion into a toast payload.

`<ToastRegion />` mounts once at editor bootstrap, alongside the existing plugin registration in `src/index.js`. No component-level changes to the recommendation panels.

### Undo path

Each toast carries the activity entry id. On Undo click, `<ToastRegion>` dispatches a thin `undoToastAction(toastId)` thunk that:

1. Calls the canonical `undoActivity(entryId)` thunk (same path `AIActivitySection` uses).
2. On resolve: dispatches `dismissToast(toastId)`.
3. On reject: dispatches `updateToast(toastId, { variant: 'error', title: 'Undo failed', detail: <original detail>, errorHint: <reason>, autoDismissMs: 8000 })`. The toast component re-renders from the new props; nothing inside the component decides "did undo fail" — that's a store fact.

This keeps the component pure-presentational and makes the variant transition observable in the store (testable via redux state assertions, recoverable across remounts).

## Risks and mitigations

- **Risk:** apply succeeds but the activity entry id is missing (e.g., when the activity write itself fails downstream). **Mitigation:** Toast still appears; Undo is disabled; user can still see the change happened. Activity-log fallback is unchanged.
- **Risk:** the user applies many recommendations in quick succession and the toast queue churns past their reading speed. **Mitigation:** non-interacted FIFO eviction protects the toast the user is actively reading or about to act on; the canonical record always stays in `Settings > AI Activity`. The toast is opportunistic, not authoritative.
- **Risk:** apply failures (already shown inline in the recommendation panel) might be expected to surface in the toast region too. **Mitigation:** explicit non-goal — the panel error is the canonical surface. A toast for the same failure would be duplicate noise. If user research later shows the inline error is missed, a `warning` variant trigger can be added in one place (`buildToastForActivity`).
- **Risk:** the body-portal mount conflicts with the Site Editor's iframed canvas. **Mitigation:** the toast region mounts to the host `document.body`, outside the iframe, where every editor surface (post editor, Site Editor, Style Book) can see it. The Inspector kit demonstrates this same placement.
- **Risk:** WP core later adds its own snackbar wrapper that visually overlaps. **Mitigation:** `z-index: 1000` and bottom placement match WP core's snackbar; if a conflict arises later, `<ToastRegion>` can adopt a higher z-index in one place.
- **Risk:** keyboard-only users on Site Editor / Style Book iframe surfaces cannot Tab into a host-body overlay. **Mitigation:** the `mod+shift+z` global shortcut moves focus directly to the newest toast's Undo button. Documented in the user-facing keyboard help via `wp.keyboardShortcuts.registerShortcut` so the shortcut is discoverable.
- **Risk:** the global `mod+shift+z` shortcut conflicts with another plugin's binding or a future WP core mapping. **Mitigation:** registered through `@wordpress/keyboard-shortcuts` (the canonical registry), so conflicts are reportable via the help modal and reassignable. The shortcut is a convenience layer; Undo is also reachable by clicking the toast directly with a pointer.

## Out of scope (explicit)

- No changes to `Activity\Repository`, `Activity\Permissions`, or `Activity\Serializer`.
- No new REST routes.
- No new abilities.
- No changes to existing surface panels (`BlockRecommendationsPanel`, `TemplateRecommender`, etc.).
- No changes to apply-action return contracts. The toast enqueue is a side effect inside the apply action, not a callsite-visible result.
- No changes to `useSuggestionApplyFeedback` or other consumers of apply-return booleans.
- No persistent toast preferences.
- No notification deduplication beyond the FIFO cap.
