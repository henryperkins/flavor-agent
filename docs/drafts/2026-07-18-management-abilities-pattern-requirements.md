## Comment body

Following the plan to establish one management ability as the agreed pattern and then replicate it: I've spent this year building against the Abilities API in a plugin whose purpose is governed mutation of a live site — 35 registered abilities, including four `request-*-apply` lanes where an external agent proposes a change and WordPress executes it only after review ([henryperkins/flavor-agent](https://github.com/henryperkins/flavor-agent)). Five contract seams turned out to be load-bearing for mutation abilities specifically. Each is cheap to settle in the first pattern and expensive to retrofit across N replicated abilities. None requires the Abilities API to ship a governance product; they define truthful contracts and attachment points that approval, exposure, audit, and undo layers can use.

**1. Keep consequence metadata composable, truthful, and homogeneous per ability.** Core already defines three independent annotations — `readonly`, `destructive`, and `idempotent` — and explicitly says registrant annotations are not guaranteed to describe behavior faithfully ([`WP_Ability` reference](https://developer.wordpress.org/reference/classes/wp_ability/)). The management pattern should specify those booleans rather than replace them with a flat read-only/additive/destructive/reversible enum. If `reversible` is added, it should be orthogonal and true only when every supported invocation has a concrete, bounded inverse or undo contract; "not destructive" does not imply reversible.

Annotations are static per ability, so they must describe the worst consequence permitted by any valid invocation. That matters for a broad ability such as the proposed [`core/manage-settings`](https://github.com/WordPress/ai/pull/764), whose one static tuple (`readonly: false`, `destructive: false`, `idempotent: true`) currently covers writes to any setting exposed through `show_in_abilities`. If inputs cross materially different risk classes, split them into risk-homogeneous abilities; otherwise annotate the broad ability conservatively for the worst case. Exposure controls can then make a defensible decision while still treating annotations as claims, not authorization. The AI Plugin's current experiment-level discussion is [`WordPress/ai#863`](https://github.com/WordPress/ai/issues/863), with [`WordPress/ai#354`](https://github.com/WordPress/ai/issues/354) tracking finer-grained exposure controls.

The distinction between declaration and curation is also important for MCP. In Flavor Agent, read-only helpers opt into public MCP metadata ([`Registration.php#L2180-L2185`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Abilities/Registration.php#L2180-L2185)); write-side recommendation abilities omit that key so they are not added to the universal default server ([`Registration.php#L2121-L2135`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Abilities/Registration.php#L2121-L2135)). A separate, dedicated Flavor Agent server uses an explicit allowlist for the recommendation and governed-apply classes ([`ServerBootstrap.php#L43-L46`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/MCP/ServerBootstrap.php#L43-L46)). That is curation by a specific server, not an annotation-to-MCP mapping.

**2. Standardize a distinct, zero-write preview ability that shares the mutation payload schema.** A `dry_run` flag on the writer is not equivalent: the ability's annotations, permission callback, discovery, and exposure are static, so a read-only invocation cannot be classified or exposed separately from real writes. Reserve either a sibling convention such as `<namespace>/preview-<operation>` or relationship metadata that identifies the preview for a writer.

For update/delete, the pair should share a named `MutationPayload` schema without requiring identical top-level inputs:

```text
preview input:  { mutation: MutationPayload }
preview output: { change: CanonicalChange, expectedState: OpaqueToken }

execute input:  { mutation: MutationPayload, expectedState: OpaqueToken }
execute output: { change: AppliedChange, changeRecord: BoundedChangeRecord }
```

Preview validates the payload, resolves the target, computes the canonical would-be change and expected-state token, and guarantees zero writes. It uses the permission required to inspect that target; execute independently requires mutation permission. Flavor Agent's six `preview-recommend-*` siblings demonstrate the value of a separately discoverable, read-only preflight surface and shared context-signature machinery ([`Registration.php#L100-L132`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Abilities/Registration.php#L100-L132), [`PreviewRecommendationAbility.php`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/AI/Abilities/PreviewRecommendationAbility.php)); the core pattern should go further by making the canonical proposed change part of the preview result.

**3. Make optimistic preconditions mandatory for update/delete, and retain a final read-to-write gate.** For an operation that changes or deletes an existing subject, `expectedState` should be required, not optional. Create operations can be exempt because no prior subject exists. If WordPress needs an unconditional update path, make it an explicit, separately permission-gated mode rather than the default produced by omitting the token.

The token should be opaque and bound to the target identity plus the bounded, relevant subject slice the preview/read observed. Execute re-resolves that slice and compares it before writing. For a batch or multi-target mutation, all preconditions must pass before the first write; any mismatch returns one standardized machine-readable error such as `ability_precondition_failed` and performs zero writes. The PHP error code should be contractual; a REST adapter can map it to conflict/precondition HTTP semantics without clients parsing human text.

There are two different race windows to close. Flavor Agent captures a live baseline when the proposal is made ([`ApplyAbilities.php#L296-L303`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Abilities/ApplyAbilities.php#L296-L303)), stores it with the pending request ([`ApplyAbilities.php#L374-L384`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Abilities/ApplyAbilities.php#L374-L384)), and compares it with live state at approval/execution ([`PendingApplyDecision.php#L100-L113`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Apply/PendingApplyDecision.php#L100-L113)). Separately, the executor re-reads immediately before persistence to close the intra-execution read→write window ([`TemplatePartApplyExecutor.php#L89-L97`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Apply/TemplatePartApplyExecutor.php#L89-L97), [`#L244-L255`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Apply/TemplatePartApplyExecutor.php#L244-L255)). The first protects proposal-to-execution freshness; the second is a final TOCTOU gate. The pattern needs both.

**4. Define a bounded change record, not universal full-subject snapshots.** A successful management execution should produce enough structured change data for review, audit, and an optional undo layer, but should not require copying the whole subject. Limit the record to the target reference, operation, and allowlisted fields actually read or changed, with bounded before/after values or hashes as the use case requires. Omit or redact secrets and personal data by default. Any persistence layer must also define per-record and per-field size limits, retention/deletion policy, who may read the record, and whether less-privileged proposal callers receive a reduced result.

`reversible: true` is justified only when the bounded record contains every value needed for a complete supported inverse, an undo path exists, and undo checks that live state still matches the recorded post-execution state before restoring anything. Flavor Agent persists structured before/after fields ([`Serializer.php#L239-L240`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Activity/Serializer.php#L239-L240), subject to the plugin's own data-handling policy) and its template-part undo refuses to revert when the current state has drifted ([`TemplatePartApplyExecutor.php#L152-L190`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Apply/TemplatePartApplyExecutor.php#L152-L190)). Without that complete bounded inverse, the annotation must remain false or unset.

**5. Make propose and execute two explicit authorization decisions.** `permission_callback` answers whether this caller may execute this ability now. A governed mutation has a separate question: may this caller enqueue this proposed change for another principal to review? Queueing is itself a durable mutation, so the proposal operation needs its own input validation, consequence annotations, permission check, rate/retention controls, and audit attribution. Approval/execution then needs a separate permission check against the live target and current actor; permission to propose must never imply permission to execute.

Two implementations fit that contract. The clearer default is a separate `propose-*`/`request-*` ability plus a non-public executor or privileged approval endpoint. If approval is mandatory, the direct writer must be absent from REST/MCP/discovery or must always deny unmediated execution. Alternatively, a substrate can mediate the writer through the `wp_pre_execute_ability` lifecycle filter landed for WordPress 7.1 in [`wordpress-develop#11731`](https://github.com/WordPress/wordpress-develop/pull/11731) ([changeset 62397 GitHub mirror](https://github.com/WordPress/wordpress-develop/commit/e8dbd462d76a380cf2da72ceb188e8486857c81a)). Because that filter can short-circuit before normal validation and permission checks, the mediator itself must validate and authorize the proposal before it queues anything. `wp_ability_permission_result` can enforce execution policy, but neither hook removes the need for the two explicit decisions.

Flavor Agent uses the first shape: four request abilities enqueue pending records, while approval is deliberately not an ability ([`Registration.php#L199-L225`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Abilities/Registration.php#L199-L225)). A request ability has its own target permission ([`RequestTemplatePartApplyAbility.php#L23-L34`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/AI/Abilities/RequestTemplatePartApplyAbility.php#L23-L34)); the admin decision route separately requires `manage_options` plus the row's contextual mutation capability ([`Agent_Controller.php#L352-L369`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/REST/Agent_Controller.php#L352-L369), [`Permissions.php#L71-L90`](https://github.com/henryperkins/flavor-agent/blob/e6a7fb3dfdae64062f5158a51f2ab5a0fbe4b69e/inc/Activity/Permissions.php#L71-L90)).

Before replicating the first management ability, I would make these contract tests: preview performs zero writes; every update/delete rejects a missing or stale precondition with zero writes; a batch checks all targets before writing; annotations match the worst valid invocation; change records stay within their allowlist/redaction/size rules; proposal-only callers cannot reach execution through any registered transport; and approval rechecks current permission and state. If the first pattern satisfies those tests, consumers can add governance without replacing the ability. If it ships as capability-check-plus-execute, the ecosystem default becomes "permission equals governance" — a question the read-only phase never had to confront.
