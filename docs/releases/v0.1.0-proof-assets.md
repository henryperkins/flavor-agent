## Why these cannot be captured from the local stack

The repo already ships capture tooling — `npm run audit:screenshot`
(`scripts/browser-run-screenshot.js`), which drives the Cloudflare Browser
Rendering API and has `settings`, `admin`, `block-editor`, and `site-editor`
presets. It cannot be pointed at the local Docker stack, because Cloudflare
renders the page from **its** network: `http://127.0.0.1:8888` is not reachable
from there. That is the structural blocker, not a missing script.

To use it, all of the following must be true at once:

- `CLOUDFLARE_ACCOUNT_ID` and `CLOUDFLARE_API_TOKEN` are set.
- `BROWSER_RUN_DEFAULT_BASE_URL` points at a **publicly reachable** WordPress
  host running this plugin — a staging site or a tunnelled local stack.
- WordPress admin authentication is supplied via `--cookies-file`,
  `BROWSER_RUN_COOKIES_JSON`, or `--extra-headers-file`; every remaining asset
  is behind `/wp-admin/`.
- For every asset except `settings-readiness`, the staged site has working
  provider credentials in `Settings > Connectors`, because the surfaces render
  a real model response. The pattern asset additionally needs a populated
  pattern index.

## Honest characterisation of the gap

Six of seven stills are missing, and each one depends on a staged site holding
live third-party credentials. That is manual release work with real secrets; it
is not automatable from this repository and it is not something to approximate.

`README.md` currently states the release should not ship without the governance
console proof plus at least one strong native editor still. Only the first of
those two exists today. Either capture at least one editor-surface still before
tagging, or consciously relax that self-imposed gate and record the decision —
but do not treat the gate as met.
